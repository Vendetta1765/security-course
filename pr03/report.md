# ПР №3. Права доступа Linux и управление пользователями

---

## 1. Пользователи и группы

Смоделирована организация из трёх ролей с разным уровнем доступа:

| Пользователь | Группа | Роль |
|---|---|---|
| alice | developers | Администратор проекта |
| bob | developers | Разработчик |
| carol | auditors | Аудитор |


**Разбор полей `/etc/passwd`** (строка `alice:x:1001:1001::/home/alice:/bin/bash`):

| Поле | Значение | Описание |
|---|---|---|
| 1 | alice | имя пользователя |
| 2 | x | пароль (хэш лежит в /etc/shadow, здесь заглушка) |
| 3 | 1001 | UID — числовой идентификатор пользователя |
| 4 | 1001 | GID — идентификатор основной группы |
| 5 | (пусто) | GECOS — комментарий / полное имя |
| 6 | /home/alice | домашняя директория |
| 7 | /bin/bash | командная оболочка |

**`/etc/shadow`** хранит хэши паролей и параметры их срока действия. Файл имеет права `-rw-r-----` (доступ только root), чтобы обычные пользователи не могли получить хэши и подобрать пароли перебором.

## 2. Права доступа chmod / chown

Папка `/srv/project/code`, владелец `alice:developers`.



**При правах 750** (`rwxr-x---`):
- **bob** (в группе developers): **смог** выполнить `ls` (есть `r-x`), но **не смог** создать файл — у группы нет права записи `w`.
- **carol** (не в группе, роль «остальные»): получила `Permission denied` — для остальных права `0`.

**После `chmod 770`** (`rwxrwx---`):
- bob **смог** создать файл `hello.py` — у группы появилось право `w`.
- carol по-прежнему без доступа.



**Файл `q1.txt`** — права `640` (`-rw-r-----`), владелец `alice:auditors`:
- alice (владелец) — чтение и запись (`rw-`),
- группа auditors, в которой carol — только чтение (`r--`),
- остальные — нет доступа.

Поэтому carol **смогла прочитать** отчёт `cat q1.txt`.

## 3. ACL — расширенные права

Задача: дать carol доступ на чтение кода, **не меняя группу** файлов.

Команда: `setfacl -m u:carol:r-x /srv/project/code`



Результат:
- В выводе `ls -la` у папки появился знак `+` — признак наличия ACL.
- carol **смогла** выполнить `ls` (через ACL получила `r-x`), но `touch` по-прежнему даёт отказ (записи нет).
- После `setfacl -x u:carol` доступ carol снова закрыт.

**Преимущество ACL:** позволяет выдать права конкретному пользователю (carol) без включения его в группу developers и без изменения группы-владельца файлов. Стандартная модель user/group/other так не умеет.

## 4. sudo-политики

| Пользователь | Разрешено | Ограничение |
|---|---|---|
| alice | Всё (`NOPASSWD:ALL`) | Нет — полный администратор |
| bob | `/usr/bin/apt`, `/usr/bin/apt-get` | Только пакетный менеджер |
| carol | `journalctl`, `cat /var/log/*` | Только просмотр логов |



При попытке `sudo journalctl` от имени bob система отказывает — ему разрешён только apt.

**Реализованный принцип безопасности — наименьшие привилегии:** каждый пользователь получает ровно тот набор прав, который нужен для его роли, и не больше. alice имеет полный доступ для администрирования, bob и carol ограничены конкретными командами.

## 5. PAM


**`pam_unix.so`** — основной модуль проверки пароля по `/etc/shadow`, используется в `/etc/pam.d/sudo` и `/etc/pam.d/login`.

**Что означает `required`** в строке `auth required pam_unix.so`: модуль **обязан** успешно отработать для прохождения аутентификации. Если он завершится с ошибкой, доступ будет отклонён, но остальные модули в стеке всё равно выполнятся (итоговый отказ).

**Политика паролей (`pam_pwquality` в `/etc/pam.d/common-password`):** настраивает требования к сложности пароля. Чтобы добавить требование минимальной длины 12 символов, к строке с `pam_pwquality.so` добавляется параметр **`minlen=12`**.

## Ответы на контрольные вопросы

1. **`chmod 640`** — владелец читает и пишет (`rw-`), группа только читает (`r--`), остальные доступа не имеют. Читать могут владелец и члены группы.
2. **setuid** даёт исполняемому файлу права его владельца для **всех**, кто его запускает (постоянно, для конкретной программы); **sudo** даёт повышение прав по политике и под контролем (кому, какие команды, с паролем или без). Пример setuid: `/usr/bin/passwd` — запускается обычным пользователем, но меняет `/etc/shadow` от root.
3. Хэши в `/etc/shadow`, потому что `/etc/passwd` читают **все** пользователи (он нужен системе для маппинга имён/UID). Если бы хэши лежали там, любой мог бы их скопировать и перебирать. `/etc/shadow` доступен только root.
4. `sudo bash` даёт bob **полноценный root-shell** — то есть фактически безграничные права, обходя любое ограничение по командам. Это эквивалент полного root-доступа, принцип наименьших привилегий нарушается.
5. **`su`** переключает на другого пользователя, требуя **его** пароль, и даёт полную сессию; **`sudo`** выполняет отдельную команду по политике, требуя **свой** пароль, с журналированием и точным ограничением прав. sudo безопаснее: granularность, логирование, не нужно раздавать пароль root.
6. Добавить пользователя в `/etc/sudoers` запрещающую запись, например через визудо: `username ALL=(ALL) !ALL` — это явно запретит ему все sudo-команды, даже если он состоит в группе sudo.
