**Дисциплина:** Программно-аппаратная защита информации

## 1. journalctl — системный журнал

auditd установлен и запущен, статус `active (running)`.

### Что зафиксировал журнал о команде `sudo cat /etc/shadow`

Журнал `_COMM=sudo` фиксирует запись о каждом вызове sudo со следующими полями:
- **пользователь**, выполнивший команду — кто запустил sudo;
- **TTY** — терминал, с которого выполнено действие;
- **PWD** — рабочий каталог на момент запуска;
- **USER** — от чьего имени выполнялась команда (root);
- **COMMAND** — полная команда с аргументами.

То есть журнал фиксирует: кто, откуда, от чьего имени и какую именно команду выполнил с повышенными правами.

### Ошибки в системе с последней загрузки

Команда `journalctl -p err -b` показывает события уровня error и выше — в основном предупреждения сервисов при старте. Критичных событий безопасности не зафиксировано.

### Статистика входов

Команды `journalctl _COMM=sshd | grep "Accepted"` и `grep "Failed"` показывают успешные и неудачные входы по SSH. На локальной VM (вход без SSH) записей входов нет. Сессии локальных пользователей и вызовы sudo/su отражаются строками `session opened/closed`.

## 2. auditd — настройка правил

### Применённые правила (вывод `auditctl -l`)
-w /etc/passwd -p rwa -k auth-files
-w /etc/shadow -p rwa -k auth-files
-w /etc/group -p rwa -k auth-files
-w /etc/sudoers -p rwa -k priv-files
-w /etc/sudoers.d -p rwa -k priv-files
-w /etc/ssh/sshd_config -p rwa -k ssh-config
-a always,exit -F arch=b64 -S open,openat -F success=0 -k access-denied
-a always,exit -F arch=b64 -S unlink,unlinkat -k file-delete
-a always,exit -F arch=b64 -S execve -F euid=0 -F auid!=0 -F auid!=-1 -k priv-exec

### Разбор записи SYSCALL
`type=SYSCALL` означает запись о системном вызове ядра — auditd фиксирует обращение программы к ядру (открытие файла, запуск процесса и т.д.).
| Поле | Что означает |
|------|-------------|
| auid | Audit UID — ID пользователя, изначально вошедшего в систему. Не меняется при su/sudo |
| uid | Реальный UID процесса в момент вызова |
| euid | Эффективный UID — с какими правами реально выполнялась операция |
| comm | Короткое имя процесса, выполнившего системный вызов |
| exe | Полный путь к исполняемому файлу |

## 3. Расследование инцидента
### Хронология действий bob
| Действие | Результат | Источник (команда поиска) |
|---------|-----------|--------------------------|
| `cat /etc/shadow` | Отказ (success=no) | `ausearch -k access-denied` |
| `ls /root` | Отказ | `ausearch -k access-denied` |
| `cat /etc/passwd` | Успех (чтение зафиксировано) | `ausearch -k auth-files` |
| `touch /tmp/totally-not-malware.sh` | Успех (файл создан) | `ausearch --start recent \| grep totally` |
| `cat /etc/sudoers` | Отказ | `ausearch -k access-denied` |
### Как auditd помог расследованию
auditd зафиксировал каждую попытку доступа к защищённым файлам, включая **неудачные** (`success=0`). По полю **auid** удалось однозначно установить, что все действия совершил один пользователь (bob), даже несмотря на смену пользователя. Это позволило восстановить полную хронологию инцидента: какие файлы пытались прочитать, что получилось, а что было заблокировано.
### Чем auditd лучше journalctl для задач аудита безопасности
auditd работает на уровне **ядра** и фиксирует системные вызовы напрямую — приложение не может «не записать» своё действие в обход аудита. journald же собирает то, что сами программы решили залогировать. Для аудита безопасности (особенно отказов в доступе и обращений к критичным файлам) auditd даёт более полную и достоверную картину, которую сложнее подделать.

## Ответы на контрольные вопросы
1. **auditd vs journald:** journald собирает сообщения, которые программы сами решили записать; auditd перехватывает системные вызовы на уровне ядра и фиксирует даже то, что приложение скрыть не может (например, отказы в доступе). Только journald недостаточно, потому что он не видит низкоуровневые обращения и не гарантирует фиксацию НСД.
2. **auid важнее uid:** auid не меняется при `su`/`sudo`, тогда как uid после повышения привилегий становится root. auid позволяет привязать действие к конкретному вошедшему пользователю.
3. **Флаг `-k`** задаёт ключ (метку) правила, по которому потом удобно искать связанные события через `ausearch -k ключ`.
4. **`-w` vs `-a always,exit`:** `-w` (watch) — простой мониторинг файла/каталога по доступу, удобен для слежки за конкретными файлами. `-a always,exit` — гибкое правило на системные вызовы с фильтрами (по syscall, успеху, euid и т.д.), используется когда нужно отслеживать типы операций, а не конкретный файл.
5. **Удаление audit.log** уничтожает следы действий злоумышленника. Защита: пересылка логов на удалённый сервер, правило `-e 2` (неизменяемая конфигурация до перезагрузки), ограничение прав на каталог логов, контроль целостности.
6. **Сохранение правил после перезагрузки:** записать их в файл в `/etc/audit/rules.d/*.rules` и применить через `augenrules --load`; временные правила `auditctl` теряются при ребуте.
