 ПР №5. GPG, подпись коммитов и безвозвратное удаление

**Дисциплина:** Программно-аппаратная защита информации
**Тема:** 1.3 Криптографическая защита · Уничтожение остаточной информации

## 1. GPG-ключевая пара

- Тип ключа: RSA 4096 бит
- Срок действия: бессрочно
- Email ключа совпадает с email на GitHub (обязательное условие для Verified)

Публичный ключ экспортирован в текстовый (ASCII) формат командой `gpg --armor --export`.

**Что означает флаг `--armor`:** вывод ключа/данных в текстовом формате ASCII (Base64) вместо бинарного. Такой формат удобно копировать, вставлять и передавать через текстовые каналы.

---

## 2. Шифрование

Файл `secrets.txt` зашифрован собственным публичным ключом командой
`gpg --encrypt --armor --recipient <email>`. Результат — файл `secrets.txt.asc` с блоком `BEGIN PGP MESSAGE`.

**Можно ли прочитать исходные данные из зашифрованного файла:** нет. Содержимое представляет собой нечитаемый шифртекст в Base64; без приватного ключа и пароля восстановить исходный текст невозможно.

**Что произойдёт при вводе неверного пароля при расшифровке:** GPG не сможет получить доступ к приватному ключу и расшифровка завершится ошибкой (`decryption failed: No secret key` / неверная passphrase), данные не будут раскрыты.

**Почему нельзя расшифровать файл, зашифрованный для другого пользователя:** файл шифруется публичным ключом получателя, а расшифровать его можно только соответствующим приватным ключом, которого нет у других. Это принцип асимметричной криптографии.

---

## 3. Цифровая подпись

Для файла `agreement.txt` создана отделённая подпись командой `gpg --armor --detach-sign`.

- Проверка `gpg --verify` неизменённого файла выдаёт **Good signature**.
- После изменения содержимого файла та же проверка выдаёт **BAD signature**.

**Что проверяет `gpg --verify`:** что файл подписан владельцем указанного ключа и что содержимое файла не изменилось с момента подписи (целостность и подлинность).

**Почему подпись стала недействительной после изменения файла:** подпись вычисляется на основе хэша содержимого. При любом изменении файла его хэш меняется и перестаёт совпадать с хэшем, зашитым в подпись, — GPG сообщает о подделке (BAD signature). Это и есть защита от подмены.

---

## 4. Подпись коммитов

Git настроен на подпись коммитов:
- `user.signingkey` — указан KEY-ID GPG-ключа;
- `commit.gpgsign true` — все коммиты подписываются автоматически.

Публичный ключ добавлен на GitHub (Settings → SSH and GPG keys). После пуша коммит получает зелёный бейдж **Verified**.

Локальная проверка командой `git log --show-signature -1` показывает `Good signature from "<имя> <email>"`.

**Зачем проверять подпись локально, если GitHub уже показывает Verified:** GitHub проверяет подпись на своей стороне и ему приходится доверять. Локальная проверка позволяет убедиться в подлинности коммита независимо от платформы — например, при работе с зеркалами репозитория или при аудите без доступа к GitHub.

## 5. rm vs shred

| Операция | Что происходит с данными на диске | Можно восстановить? |
|----------|-----------------------------------|---------------------|
| `rm` | Удаляется только запись в метаданных (inode), блоки данных помечаются как свободные, но физически не перезаписываются | Да, пока блоки не перезаписаны другими данными |
| `shred` | Содержимое блоков несколько раз перезаписывается случайными данными и нулями, затем файл удаляется | Нет |

**Что делает `rm`:** убирает ссылку на файл из каталога и освобождает inode; само содержимое блоков остаётся на диске до перезаписи. Поэтому данные можно восстановить специальными утилитами.

**Чем `shred` отличается принципиально:** он физически перезаписывает блоки файла (в работе — 3 прохода + обнуление), уничтожая остаточную информацию, и только потом удаляет файл.

**Сценарий, когда нужно затирать свободное место:** перед списанием или продажей носителя, на котором ранее хранились конфиденциальные данные и файлы уже были удалены обычным `rm`. Затирание свободного места уничтожает остатки этих файлов, которые иначе можно восстановить.

## Ответы на контрольные вопросы

1. **Шифрование vs подпись:** шифрование скрывает содержимое (конфиденциальность), подпись подтверждает автора и целостность (аутентичность). Да, файл можно одновременно зашифровать и подписать (`gpg --sign --encrypt`).
2. **Публичный ключ** служит только для шифрования данных в твой адрес и проверки твоей подписи — он безопасен для раздачи. **Приватный ключ** позволяет расшифровывать и подписывать от твоего имени, поэтому его компрометация = компрометация владельца.
3. **Потеря приватного ключа** означает, что зашифрованные им данные невозможно расшифровать — они теряются безвозвратно.
4. **Подделать Verified нельзя** без приватного GPG-ключа владельца: бейдж появляется только если коммит подписан ключом, привязанным к аккаунту.
5. **shred плохо работает на SSD** из-за wear-leveling: контроллер пишет в другие физические ячейки, и перезапись не затирает оригинал. Вместо него — встроенная команда ATA Secure Erase, полное шифрование диска (LUKS) или аппаратное стирание.
6. **Форматирование не гарантирует безопасность:** быстрое форматирование лишь пересоздаёт файловую систему, данные остаются в блоках и восстанавливаются. Коллега не прав — нужно полное затирание или шифрование.
7. **`--detach-sign`** создаёт отдельный файл подписи (оригинал остаётся как есть); **`--sign`** встраивает подпись в сам файл, упаковывая данные и подпись вместе.
